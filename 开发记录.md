# 项目开发记录

## 1. 记录说明

本文档基于当前项目代码、技术规范文档和可行性报告整理，用于说明项目实现过程、关键设计决策、当前落地范围和后续建议。

## 2. 项目目标回顾

项目目标是将 Dify 教学智能体接入自研教学管理系统，实现：

- 学生统一登录与对话学习
- 教师对班级和学生进行管理
- 聊天日志沉淀到本地数据库
- 自动评分与教师复核
- 管理员统一维护 Dify 平台配置

从当前代码看，核心目标已经落地，项目已形成可运行的前后端系统，而不是仅停留在方案设计阶段。

## 3. 当前实现架构

### 3.1 后端

后端基于 `Django 4.2 + DRF + SimpleJWT`，主要应用如下：

- `apps.users`：用户、专业、班级、管理员接口、教师接口
- `apps.dify_integration`：Dify 配置、聊天代理、工作流、日志同步
- `apps.chat_logs`：会话与聊天日志
- `apps.scoring`：评分模型、评分引擎、教师统计与复核

### 3.2 前端

前端基于 `Vue 3 + Vite + Pinia + Vue Router + Element Plus`，已实现：

- 登录页
- 学生聊天页
- 教师端管理页
- 管理员端管理页

### 3.3 路由结构

当前路由体现了明确的角色分流：

- 学生：`/chat`
- 教师：`/teacher/dashboard` 等
- 管理员：`/admin/dashboard` 等

路由守卫按 `localStorage` 中的 token 和角色判断访问权限。

## 4. 关键开发阶段

### 4.1 第一阶段：基础后端框架与认证

完成内容：

- Django 项目初始化
- 用户模型扩展
- JWT 登录认证
- 获取个人信息
- 修改密码

设计结果：

- 用 `User.role` 表示角色
- 用 `must_change_password` 支持首次登录改密

### 4.2 第二阶段：组织结构模型

完成内容：

- 专业模型 `Major`
- 班级模型 `SchoolClass`
- 学生归属班级
- 教师与班级绑定

设计原因：

- 业务要求不再是扁平用户管理，而是“教师管理班级，班级管理学生”
- 使用 `PROTECT` 可以避免误删导致关联数据失真

### 4.3 第三阶段：Dify 聊天代理接入

完成内容：

- 后端代理聊天请求
- SSE 流式返回前端
- 会话与消息日志入库
- 文件上传
- 停止生成
- 历史同步入口

设计原因：

- 不直接让前端暴露 Dify API Key
- 通过后端代理，才能把对话行为沉淀到本地数据库
- 流式输出提升学生端交互体验

### 4.4 第四阶段：教师管理能力

完成内容：

- 教师管理专业
- 教师管理班级
- 教师管理学生
- 学生密码重置
- CSV 严格事务导入

这里有一个明显的实现特征：

- 教师不仅管理班级和学生，还具备专业管理能力

这与部分规划文档中“专业仅由管理员管理”的表述存在差异，最终应以当前代码实现为准。

### 4.5 第五阶段：管理员后台能力

完成内容：

- 管理员管理教师账号
- 教师停用
- 教师密码重置
- Dify 配置单例管理

设计结果：

- 管理员角色被收敛为“平台与教师管理者”
- Dify 配置由数据库单例承载，支持运行期修改

### 4.6 第六阶段：评分引擎与统计

完成内容：

- 会话评分模型
- 评分明细模型
- 自动评分
- 教师复核
- 统计看板
- 批量重算

当前评分核心维度：

- 阶段完成度
- 流程遵循度
- 时间投入
- 参与度

## 5. 关键设计决策

### 5.1 使用数据库单例承载平台配置

相关模型：

- `DifyConfig`
- `ScoringConfig`

决策原因：

- 避免每次改配置都修改 `.env`
- 支持前端可视化配置
- 修改后即时生效

### 5.2 学生与 Dify 用户一对一映射

相关模型：

- `DifyUserMapping`

决策原因：

- Dify 需要稳定的 `user` 标识
- 学生历史会话、日志同步、文件上传都依赖这层映射

### 5.3 评分与展示共用阶段消息门槛

相关模型：

- `WorkflowStage.expected_min_messages`

决策原因：

- 统一“阶段完成”的判断口径
- 教师修改门槛后，看板和评分逻辑同时联动

### 5.4 使用严格事务处理 CSV 导入

决策原因：

- 避免部分数据成功、部分失败导致班级名单不一致
- 教师可以直接根据错误信息修复后重新导入

## 6. 当前实现与规划文档的差异

结合代码与文档，可见以下实际差异：

### 6.1 专业管理权限

文档中有“管理员维护专业字典”的表述，但当前实现里：

- 教师端存在 `teacher_majors_manage_view`
- 路由中存在 `teacher/majors`

因此当前系统实际上允许教师管理专业。

### 6.2 管理员页面范围

技术规范文档中列出了管理员专业管理页，但当前前端管理员路由实际只有：

- `admin/dashboard`
- `admin/teachers`
- `admin/dify-config`

管理员专业管理页面当前并未在前端落地。

### 6.3 评分最小时长默认值

规划文档中曾写为 `5` 分钟，但当前代码默认值为：

- `SCORING_CONFIG.MIN_SESSION_MINUTES = 1`
- `ScoringConfig.min_session_minutes` 默认也是 `1`

因此安装和使用时应以代码为准。

## 8. 已知限制

### 8.1 SQLite 适用范围有限

当前默认数据库是 SQLite，适合开发测试，不适合高并发生产场景。

### 8.2 流式接口依赖 ASGI

开发环境 `runserver` 可用，但正式环境若要稳定支持 SSE，应部署为 ASGI。

### 8.3 阶段识别依赖 Dify 工作流配合

若 Dify 工作流未正确输出阶段信息，系统依然能聊天，但阶段统计和评分会受影响。

### 8.4 测试覆盖度有限

仓库中虽有 `tests.py`，但从当前结构看，项目主要价值仍体现在功能落地，自动化测试覆盖度不高。
