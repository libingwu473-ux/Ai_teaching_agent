## 1. 项目概述与技术栈

### 1.1 项目背景

教师在 Dify 平台上设计了一个 **多阶段 AI 教学智能体**（类型：Agent，含知识库RAG），学生通过与智能体对话完成学习。本项目构建一个中间层系统，实现用户管理、对话代理、行为追踪、自动评分和教学分析。

### 1.2 核心目标

| 序号 | 目标 | 说明 |
|------|------|------|
| 1 | 用户绑定 | 我方注册用户与 Dify 用户 ID 一对一映射 |
| 2 | 对话代理 | 通过自有后端代理 Dify 对话 API，支持流式(SSE)响应 |
| 3 | 日志记录 | 完整记录每次对话的提问、回答、token 用量、时间、所处阶段 |
| 4 | 阶段追踪 | 识别学生当前所处的学习阶段，判断是否按教师设计流程进行 |
| 5 | 自动评分 | 根据阶段完成度、时间投入、对话质量自动生成评分 |
| 6 | 教师仪表盘 | 可视化展示学生学习数据，支持教师人工复核评分 |

### 1.3 技术栈

### 2.2 数据流设计

```
【注册流程】
用户注册 → Django创建User → 生成dify_user_id → 存入DifyUserMapping

【对话流程】
前端发起聊天 → Django接收 → 转发到Dify API(流式) →
同时: 解析SSE → 提取阶段信息 → 写入ChatLog →
流式返回前端(逐字渲染)

【评分流程】
教师触发/定时任务 → 查询已完成会话 → 提取日志数据 →
计算阶段完成度/时间/轮次 → 生成自动评分 →
标记待教师复核

【教师复核流程】
教师登录 → 查看学生列表 → 点击某个学生 →
查看该学生所有会话 → 查看AI自动评分 →
教师修改/确认评分 → 保存最终评分
```

## 4. 用户注册与 Dify 用户绑定流程

### 4.1 核心流程

```
┌──────────┐     ┌──────────┐     ┌──────────────────┐
│ 用户注册  │ ──→ │ 创建User │ ──→ │ 生成dify_user_id │
│(邮箱+密码)│     │ (Django) │     │ (UUID格式)        │
└──────────┘     └──────────┘     └────────┬─────────┘
                                           │
                                           ▼
                                  ┌──────────────────┐
                                  │ DifyUserMapping  │
                                  │ user_id: 1       │
                                  │ dify_user_id:    │
                                  │ dify_edu_xxxxxxxx│
                                  │ is_active: true  │
                                  └──────────────────┘
```

### 4.2 关键设计

- **dify_user_id 格式**: `dify_edu_{uuid4().hex[:12]}`, 全平台唯一
- **自动生成**: 通过 Django Signal (`post_save`) 自动创建映射记录
- **用户角色**: `student`（学生）、`teacher`（教师）、`admin`（管理员）
- **认证**: 使用 `djangorestframework-simplejwt` 的 JWT Token

## 5. Dify API 对接规范

### 5.1 认证方式

```
Authorization: Bearer app-wdCJUYFNUsRUqoH9dWyE0BYO
Content-Type: application/json
```

### 5.2 已实现的核心 API 端点

| 端点 | 方法 | 用途 | 我方代理 |
|------|------|------|----------|
| `/v1/chat-messages` | POST (SSE) | 发送对话消息(流式) | `/api/chat/send/` |
| `/v1/chat-messages/{task_id}/stop` | POST | 停止生成 | `/api/chat/stop/` |
| `/v1/conversations` | GET | 获取会话列表 | `/api/chat/conversations/` |
| `/v1/messages` | GET | 获取会话消息历史 | 同步服务使用 |
| `/v1/conversations/{id}` | DELETE | 删除会话 | `/api/chat/conversations/{id}/delete/` |
| `/v1/files/upload` | POST | 上传文件 | `/api/chat/upload/` |

### 5.3 发送消息请求参数

```json
{
    "query": "什么是机器学习？",
    "inputs": {},
    "response_mode": "streaming",
    "conversation_id": "",
    "user": "dify_edu_a1b2c3d4e5f6",
    "files": [],
    "auto_generate_name": true
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `query` | string | ✅ | 用户问题 |
| `inputs` | object | ✅ | 应用输入变量（可为空对象） |
| `response_mode` | string | ✅ | 固定 `streaming` |
| `conversation_id` | string | ❌ | 空字符串=新建会话 |
| `user` | string | ✅ | 用户的 dify_user_id |

### 5.4 Agent 类型 SSE 事件参考

本智能体为 **Agent** 类型，SSE 事件与 Chatflow 类型不同：

| 事件 | 说明 | 关键字段 |
|------|------|----------|
| `agent_thought` | Agent 思考过程 | `conversation_id`, `message_id`, `thought` |
| `agent_message` | Agent 回复片段(流式) | `answer` (增量文本) |
| `message_end` | 消息结束 | `conversation_id`, `metadata.usage` |
| `error` | 错误 | `message`, `code` |

**关键差异**: Agent 类型使用 `agent_message` 而非 `text_chunk`，token 用量在 `message_end.metadata.usage` 中。

### 5.5 获取会话消息历史

```
GET /v1/messages?user={dify_user_id}&conversation_id={conv_id}&limit=100
```

**响应**:
```json
{
    "limit": 100,
    "has_more": false,
    "data": [
        {
            "id": "message-uuid",
            "conversation_id": "conv-uuid",
            "query": "什么是机器学习？",
            "answer": "机器学习是人工智能的一个分支...",
            "inputs": {"current_stage": "stage_concept"},
            "created_at": 1705407629
        }
    ]
}
```

### 5.6 错误码处理

| HTTP状态码 | 说明 | 处理方式 |
|-----------|------|----------|
| 200 | 成功(流式) | 正常处理SSE流 |
| 301 | HTTP→HTTPS 重定向 | 使用 HTTPS URL |
| 400 | 参数错误 | 检查请求参数 |
| 401 | API Key 无效 | 检查配置 |
| 429 | 速率限制 | 等待后重试 |
| 500 | Dify 内部错误 | 记录日志，通知用户 |

## 8. 学习阶段追踪与评分引擎

### 8.1 阶段追踪策略

#### 策略A: Dify 会话变量（主要方式 — 需教师配置）

教师在 Dify Workflow 中配置：
1. 使用「变量赋值器(Variable Assigner)」节点
2. 每个阶段节点执行完毕后，更新会话变量 `current_stage`
3. 将已完成阶段追加到 `completed_stages` 数组变量

**Dify 端配置**:
```
Workflow:
  [开始] → [概念讲解节点] → [Variable Assigner: current_stage="stage_concept"] 
        → [练习测验节点] → [Variable Assigner: current_stage="stage_practice"]
        → [总结评估节点] → [Variable Assigner: current_stage="stage_summary"]
        → [结束]
```

#### 策略B: SSE 事件解析（辅助方式）

后端解析 SSE 流中的 `node_started` / `node_finished` / `workflow_finished` 事件，从 `outputs` 字段中提取 `current_stage` 变量。

### 8.2 评分引擎算法

#### 8.2.1 阶段完成度评分 (权重 40%)

```
score_stage = (完成阶段数 / 总阶段数) × 100 × 0.40
```

- 阶段"完成"判定：该阶段至少有1条用户消息
- 按顺序完成额外加分（最多 +10 分）

#### 8.2.2 流程遵循度评分 (权重 25%)

```
score_sequence = (按顺序完成的阶段数 / 总阶段数) × 100 × 0.25
```

- 检查阶段访问顺序是否与预期一致
- 跳过阶段扣分，重复阶段不影响

#### 8.2.3 时间投入评分 (权重 15%)

```
score_time = min(实际耗时 / 预期总耗时, 1.0) × 100 × 0.15
```

- 实际耗时 = session.ended_at - session.started_at
- 低于 MIN_SESSION_MINUTES(5分钟) 直接 0 分
- 超过预期时间不额外加分(上限 1.0)

#### 8.2.4 参与度评分 (权重 20%)

```
score_engagement = min(总消息数 / 预期总消息数, 1.0) × 100 × 0.20
```

#### 8.2.5 总分计算

```
auto_total_score = stage_completion + sequence_adherence + time_investment + engagement
满分: 100分
```

**评分等级**:
| 分数 | 等级 |
|------|------|
| 90-100 | 优秀 |
| 80-89 | 良好 |
| 70-79 | 中等 |
| 60-69 | 及格 |
| <60 | 需改进 |

### 8.3 评分流程

```
教师/定时触发 → 获取会话logs → 获取workflow stages →
┌─ 计算阶段完成度 (40%)
├─ 计算流程遵循度 (25%)
├─ 计算时间投入 (15%)
└─ 计算参与度 (20%)
    ↓
生成 StudentScore + ScoreDetail →
标记 status=pending_review →
教师可在仪表盘查看/修改 → 最终确认
```

### 9.1 设计目的

### 9.2 同步逻辑

```python
sync_all_active_users():
  for each active DifyUserMapping:
    1. 从 Dify API 获取该用户的会话列表 (GET /v1/conversations)
    2. 对比本地 ConversationSession 表，创建新会话记录
    3. 对每个会话，获取消息历史 (GET /v1/messages)
    4. 对比本地 ChatLog 表，创建新消息记录
    5. 更新会话统计（总消息数、总 token、阶段状态）
```

```javascript
import { fetchEventSource } from '@microsoft/fetch-event-source'

async function sendMessage() {
  await fetchEventSource('/api/chat/send/', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${token}`,
    },
    body: JSON.stringify({ query, conversation_id: currentConvId || '' }),
    signal: controller.signal,

    onmessage(event) {
      const data = JSON.parse(event.data)

      // Agent 类型应用使用 agent_message 事件
      if (event.event === 'agent_message' && data.answer) {
        streamingText += data.answer
      }
      // Chatflow 类型使用 text_chunk 事件
      else if (event.event === 'text_chunk') {
        streamingText += data.data || data.text || ''
      }
      // 阶段变化
      else if (event.event === 'stage_change') {
        currentStage = data.stage
      }
      // 消息完成
      else if (event.event === 'message_end') {
        conversationId = data.conversation_id
        // 完成流式渲染
      }
    },
  })
}
```

## 12. 开发与部署注意事项

### 12.1 流式模式特殊处理

1. **Django ASGI**: 生产环境必须使用 ASGI 服务器(Daphne/Uvicorn)，WSGI 不支持流式响应
2. **httpx 流式**: 必须使用 `client.stream()` 上下文管理器，不可用 `.json()` 等方法
3. **Agent vs Chatflow**: Agent 类型使用 `agent_message` 事件，Chatflow 使用 `text_chunk`
4. **连接中断**: 前端需处理 SSE 意外断开，后端需捕获 `httpx.RemoteProtocolError`

### 12.2 HTTPS 注意事项

1. Dify API 使用 HTTPS (`https://dify.chat.43d.cn/v1`)
2. HTTP 请求会被 301 重定向到 HTTPS
3. httpx 默认不自动跟随 POST 重定向，需使用 HTTPS URL

### 12.3 SQLite 限制

1. **并发写入**: SQLite 不支持高并发，多用户同时对话可能出现 `database locked`
2. **生产环境建议**: 切换到 PostgreSQL 或 MySQL
3. **开发环境**: SQLite 足够用于开发和测试

### 12.4 安全清单

- [ ] API Key 不暴露到前端
- [ ] JWT Token 有效期合理设置
- [ ] CORS 白名单配置
- [ ] 接口频率限制（建议 30次/分钟/用户）
- [ ] 生产环境 SECRET_KEY 更换
- [ ] DEBUG=False

### 12.5 教师 Dify 配置指南

为了让系统正确追踪学习阶段，教师需要在 Dify Workflow 中做以下配置：

1. **初始化会话变量**：
   - `current_stage`: String, 初始值 `"stage_concept"`
   - `completed_stages`: Array[String], 初始值 `[]`

2. **每个阶段结束时**添加「变量赋值器」节点：
   - Overwrite `current_stage` 为当前阶段的 key
   - Append `current_stage` 到 `completed_stages` 数组

3. **阶段命名建议**：
   - `stage_concept` — 概念讲解
   - `stage_practice` — 练习测验
   - `stage_summary` — 总结评估
