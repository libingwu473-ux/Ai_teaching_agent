## 1. 项目概述与技术栈

### 1.1 项目背景

教师在 Dify 平台上设计了一个 **多阶段 AI 教学智能体**（类型：Agent，含知识库RAG），学生通过与智能体对话完成学习。本项目构建一个中间层系统，实现用户管理、对话代理、行为追踪、自动评分和教学分析。

### 1.2 核心目标

| 序号 | 目标 | 说明 |
|------|------|------|
| 1 | 三级用户层级 | admin 管理教师与平台配置；teacher 管理班级与学生；student 在所属班级下学习 |
| 2 | 用户绑定 | 学生用户与 Dify 用户 ID 一对一映射（teacher/admin 不生成 Dify 映射） |
| 3 | 对话代理 | 通过自有后端代理 Dify 对话 API，支持流式(SSE)响应 |
| 4 | 日志记录 | 完整记录每次对话的提问、回答、token 用量、时间、所处阶段 |
| 5 | 阶段追踪 | 识别学生当前所处的学习阶段，判断是否按教师设计流程进行 |
| 6 | 自动评分 | 根据阶段完成度、流程遵循度、时间投入、参与度自动生成评分 |
| 7 | 教师仪表盘 | 可视化展示学生学习数据，支持教师人工复核评分 |
| 8 | 管理员后台 | 教师 CRUD、专业字典管理、Dify 平台配置统一入口 |
| 9 | 批量录入 | CSV 严格事务导入学生（学号/姓名必填，密码留空默认 = 学号） |

## 2. 系统架构设计

### 2.1 用户层级模型（v2.0 新增）

```
admin (超级管理员，单例)
  ├── 管理 DifyConfig (API 地址 / API Key / SSL / 超时 / 重试)
  ├── 管理 Major (专业字典)
  └── 管理 Teacher 账号（增/改/重置密码/停用）
        │
        └── Teacher (教师)
              └── 管理 SchoolClass (班级 = 专业 + 班级名 + 唯一教师)
                    │  一个教师可管理多个班级；一个班级仅属一个教师
                    │
                    └── Student (学生，必须挂在某个班级下)
                          └── 通过 ChatView 与 Dify 对话；产生 ConversationSession + ChatLog
```

层级约束：
- 一个教师可拥有多个 `SchoolClass`；一个 `SchoolClass.teacher` 仅指向一个 teacher。
- `SchoolClass(major, name)` 在专业下唯一。
- 学生的 `school_class` 使用 `ForeignKey(PROTECT)`，避免班级被误删带走学生数据。
- admin 不直接 CRUD 学生（只 teacher 负责）。

### 2.3 数据流设计

```
【账号创建流程】（v2.0 改为自顶向下）
seed_admin → 创建 admin/admin123 → admin 登录 → 创建 Teacher 账号
                                          → 维护 Major 专业字典
                                          → 配置 Dify 平台连接

【对话流程】（与 v1.0 一致）

教师/定时触发 → 查询已完成会话 → 提取日志数据 →

计算阶段完成度/流程遵循度/时间/参与度 →
生成自动评分 → 标记待教师复核

【教师复核流程】
教师登录 → 仪表盘 → 学生列表 → 学生详情 →
查看会话与自动评分 → 复核页修改 teacher_score / comment / status → 保存
```

### 3.3 安全注意事项

1. **API Key 保护**: `DIFY_API_KEY` 仅初次启动作为默认值；运行后由 `DifyConfig` 数据库单例托管。前端读取时 `api_key` 默认被掩码（如 `app-wd...0BYO`），仅 admin 可显式 `?reveal=1` 拉取明文。
2. **JWT Token**: Access 30 分钟，Refresh 7 天。
3. **CORS**: 已为 `localhost:5173` 开启凭据；生产环境需配置白名单域名。
4. **HTTPS**: Dify API 使用 HTTPS，注意 httpx 的 301 重定向处理。
5. **角色守卫**: 后端每个 admin/teacher 接口都显式校验 `request.user.role`，前端 router 守卫只做用户体验，不能作为安全防线。
6. **公开注册关闭**: `/api/auth/register/` 已删除；账号只能由上级创建，避免外部注册风险。
7. **默认密码**: `seed_admin` 创建 `admin/admin123` 时 `must_change_password=True`，首次登录建议立即改密；学生默认密码 = 学号，同样设置该标志。

## 4. 用户层级与账号创建流程

### 4.1 三级账号创建链（v2.0 新增）

```
[1] seed_admin (CLI)
    └─► admin / admin123 (role=admin, is_superuser=True, must_change_password=True)

[2] admin 在 /admin/teachers 新建
    └─► teacher 账号 (role=teacher, must_change_password=True)
         可选：admin 在 /admin/majors 维护专业字典

[3] teacher 登录 /teacher/classes 新建班级
    └─► SchoolClass(major_id, name, teacher=request.user)

[4] teacher 在 /teacher/classes/<id>/students 录入
    单条录入  ─► User(role=student, school_class=该班级, must_change_password=True)
    批量导入  ─► CSV 严格事务（学号/密码/专业/班别/姓名/性别）
```

公开注册接口 `/api/auth/register/` 已删除。

### 4.2 Dify 用户绑定（仅 student）

```
[teacher 录入 / CSV 导入] → User(role=student) 保存 → post_save 信号触发
                                                  → 生成 dify_edu_<uuid12>
                                                  → 创建 DifyUserMapping
admin / teacher 角色 → 不触发 DifyUserMapping（无需 Dify 身份）
```

- **dify_user_id 格式**：`dify_edu_{uuid4().hex[:12]}`，全平台唯一
- **触发机制**：[apps/users/signals.py](backend/apps/users/signals.py) 的 `post_save` 信号，仅当 `instance.role == 'student'`
- **存量学生清理**：[users/migrations/0003_purge_students.py](backend/apps/users/migrations/0003_purge_students.py) 在迁移时清空所有 `role=student` 用户及其级联数据（ChatLog/StudentScore/ScoreDetail/DifyUserMapping）

### 4.3 角色与权限矩阵

| 接口前缀 | role=admin | role=teacher | role=student | 备注 |
|----------|-----------|-------------|-------------|------|
| `/api/auth/login,profile,change-password/` | ✅ | ✅ | ✅ | 全部已认证用户 |
| `/api/admin/*` | ✅ | ❌ | ❌ | 仅 admin |
| `/api/teacher/classes,class-students,majors-readonly/` | 只读穿透 | ✅（仅自己班级）| ❌ | teacher 写，admin 只能 GET |
| `/api/teacher/students,scores,stats,scoring/*` | ✅ | ✅ | ❌ | scoring 模块的"全局教师视图"，admin 也可访问 |
| `/api/chat/send,stop,conversations,upload/` | ✅ | ✅ | ✅ | 学生主用；admin/teacher 也可调试 |

| `user` | string | ✅ | 用户的 dify_user_id |

### 5.4 Agent 类型 SSE 事件参考

本智能体为 **Agent** 类型，SSE 事件与 Chatflow 类型不同：

| 事件 | 说明 | 关键字段 |
|------|------|----------|
| `agent_thought` | Agent 思考过程 | `conversation_id`, `message_id`, `thought` |
| `agent_message` | Agent 回复片段(流式) | `answer` (增量文本) |
| `message_end` | 消息结束 | `conversation_id`, `metadata.usage` |
| `error` | 错误 | `message`, `code` |

**关键差异**: Agent 类型使用 `agent_message` 而非 `text_chunk`，token 用量在 `message_end.metadata.usage` 中。

## 8. 学习阶段追踪与评分引擎

### 8.1 阶段追踪策略

#### 策略A: Dify 会话变量（主要方式 — 需教师配置）

教师在 Dify Workflow 中配置：
1. 使用「变量赋值器(Variable Assigner)」节点
2. 每个阶段节点执行完毕后，更新会话变量 `current_stage`
3. 将已完成阶段追加到 `completed_stages` 数组变量

**Dify 端配置**:
```
Workflow:
  [开始] → [概念讲解节点] → [Variable Assigner: current_stage="stage_concept"] 
        → [练习测验节点] → [Variable Assigner: current_stage="stage_practice"]
        → [总结评估节点] → [Variable Assigner: current_stage="stage_summary"]
        → [结束]
```

#### 策略B: SSE 事件解析（辅助方式）

后端解析 SSE 流中的 `node_started` / `node_finished` / `workflow_finished` 事件，从 `outputs` 字段中提取 `current_stage` 变量。

### 8.2 评分引擎算法

#### 8.2.1 阶段完成度评分 (权重 40%)

```
对每个阶段 s:
  msg_count_s = 本会话中 ChatLog.stage_key == s.stage_key 的条数
  is_completed_s = msg_count_s >= s.expected_min_messages
score_stage_raw = (完成阶段数 / 总阶段数) × 100
score_stage     = score_stage_raw × 0.40
```

- 阶段"完成"判定：**该阶段累计消息数 ≥ 该阶段的 `expected_min_messages`**（v2.0 起明确：从"至少1条"改为按门槛配置）
- 教师可在 `/teacher/settings` 调整每个阶段的 `expected_min_messages`
- 教师仪表盘的"阶段完成率"使用相同门槛，但采用 **平均进度口径**（详见 §7.3）

#### 8.2.2 流程遵循度评分 (权重 25%)

```
score_sequence = (按顺序完成的阶段数 / 总阶段数) × 100 × 0.25
```

- 检查阶段访问顺序是否与预期一致
- 跳过阶段扣分，重复阶段不影响

#### 8.2.3 时间投入评分 (权重 15%)

```
score_time = min(实际耗时 / 预期总耗时, 1.0) × 100 × 0.15
```

- 实际耗时 = session.ended_at - session.started_at
- 低于 MIN_SESSION_MINUTES(5分钟) 直接 0 分
- 超过预期时间不额外加分(上限 1.0)

#### 8.2.4 参与度评分 (权重 20%)

```
score_engagement = min(总消息数 / 预期总消息数, 1.0) × 100 × 0.20
```

#### 8.2.5 总分计算

```
auto_total_score = stage_completion + sequence_adherence + time_investment + engagement
满分: 100分
```

**评分等级**:
| 分数 | 等级 |
|------|------|
| 90-100 | 优秀 |
| 80-89 | 良好 |
| 70-79 | 中等 |
| 60-69 | 及格 |
| <60 | 需改进 |

### 8.3 评分流程

```
教师/定时触发 → 获取会话logs → 获取workflow stages →
┌─ 计算阶段完成度 (40%)
├─ 计算流程遵循度 (25%)
```

## 9. 定时同步服务设计

### 9.1 设计目的

当用户通过 iframe 或直接使用 Dify 对话时，对话日志不会自动进入我方数据库。定时同步服务定期从 Dify API 拉取最新会话和消息数据。

### 9.2 同步逻辑

```python
sync_all_active_users():
  for each active DifyUserMapping:
    1. 从 Dify API 获取该用户的会话列表 (GET /v1/conversations)
    2. 对比本地 ConversationSession 表，创建新会话记录
    3. 对每个会话，获取消息历史 (GET /v1/messages)
    4. 对比本地 ChatLog 表，创建新消息记录
    5. 更新会话统计（总消息数、总 token、阶段状态）
```

### 9.3 触发方式

```bash
# 方式1: Django 管理命令
python manage.py sync_dify_logs

# 方式2: 指定用户
python manage.py sync_dify_logs --user-id 1

# 方式3: API 手动触发 (仅教师/管理员)
POST /api/chat/sync/

# 方式4: 定时任务 (生产环境使用 cron / Celery Beat)
*/2 * * * * cd /path/to/backend && python manage.py sync_dify_logs
```

### 10.4 流式消息处理 (关键代码)

```javascript
import { fetchEventSource } from '@microsoft/fetch-event-source'

async function sendMessage() {
  await fetchEventSource('/api/chat/send/', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${token}`,
    },
    body: JSON.stringify({ query, conversation_id: currentConvId || '' }),
    signal: controller.signal,

    onmessage(event) {
      const data = JSON.parse(event.data)

      // Agent 类型应用使用 agent_message 事件
      if (event.event === 'agent_message' && data.answer) {
        streamingText += data.answer
      }
      // Chatflow 类型使用 text_chunk 事件
      else if (event.event === 'text_chunk') {
        streamingText += data.data || data.text || ''
      }
      // 阶段变化
      else if (event.event === 'stage_change') {
        currentStage = data.stage
      }
      // 消息完成
      else if (event.event === 'message_end') {
        conversationId = data.conversation_id
        // 完成流式渲染
      }
    },
  })
}
```

## 12. 开发与部署注意事项

### 12.1 流式模式特殊处理

1. **Django ASGI**: 生产环境必须使用 ASGI 服务器(Daphne/Uvicorn)，WSGI 不支持流式响应
2. **httpx 流式**: 必须使用 `client.stream()` 上下文管理器，不可用 `.json()` 等方法
3. **Agent vs Chatflow**: Agent 类型使用 `agent_message` 事件，Chatflow 使用 `text_chunk`
4. **连接中断**: 前端需处理 SSE 意外断开，后端需捕获 `httpx.RemoteProtocolError`

### 12.2 HTTPS 注意事项

1. Dify API 使用 HTTPS (`https://dify.chat.43d.cn/v1`)
2. HTTP 请求会被 301 重定向到 HTTPS
3. httpx 默认不自动跟随 POST 重定向，需使用 HTTPS URL

### 12.3 SQLite 限制

1. **并发写入**: SQLite 不支持高并发，多用户同时对话可能出现 `database locked`
2. **生产环境建议**: 切换到 PostgreSQL 或 MySQL
3. **开发环境**: SQLite 足够用于开发和测试

### 12.4 安全清单

- [ ] API Key 不暴露到前端
- [ ] JWT Token 有效期合理设置
- [ ] CORS 白名单配置
- [ ] 接口频率限制（建议 30次/分钟/用户）
- [ ] 生产环境 SECRET_KEY 更换
- [ ] DEBUG=False

### 12.5 教师 Dify 配置指南

为了让系统正确追踪学习阶段，教师需要在 Dify Workflow 中做以下配置：

1. **初始化会话变量**：
   - `current_stage`: String, 初始值 `"stage_concept"`
   - `completed_stages`: Array[String], 初始值 `[]`

2. **每个阶段结束时**添加「变量赋值器」节点：
   - Overwrite `current_stage` 为当前阶段的 key
   - Append `current_stage` 到 `completed_stages` 数组

3. **阶段命名建议**：
   - `stage_concept` — 概念讲解
   - `stage_practice` — 练习测验
   - `stage_summary` — 总结评估
