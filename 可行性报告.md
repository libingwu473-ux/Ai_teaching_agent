## 1. 项目背景与需求分析

### 1.1 业务场景

教师利用 Dify 平台搭建了一个 **AI 教学智能体**（Agent 类型，含知识库 RAG），设计为多阶段教学对话流程：

```
概念讲解（stage_concept）→ 练习测验（stage_practice）→ 总结评估（stage_summary）
```

学生在上述三个阶段依次与智能体对话学习。教育管理者需要：

1. **追踪学习行为** — 学生是否按顺序完成各阶段
2. **记录学习数据** — 提问内容、回答、对话轮次、Token 消耗、时间投入
3. **自动评估** — 根据预设规则自动打分
4. **人工复核** — 教师查看系统评分并最终确认

### 1.2 核心需求清单

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 三级用户层级 | P0 | admin 管教师与平台配置；teacher 管班级与学生；student 在班级下学习 |
| 班级化管理 | P0 | 学生必须挂在某个班级，专业 → 班级 → 学生三级字典 |
| 教师独占其管理范围 | P0 | 一个班级仅属一个教师；教师只能 CRUD 自己班级的学生 |
| 用户绑定 Dify | P0 | 仅学生注册时一对一映射 Dify user 参数 |
| 流式对话代理 | P0 | 通过自有后端代理 Dify SSE 流，同时记录日志 |
| 对话日志持久化 | P0 | 完整记录每次对话的提问、回答、Token、时间 |
| 学习阶段追踪 | P1 | 识别学生所处阶段，判断是否按流程进行 |
| 自动评分引擎 | P1 | 多维度加权评分（完成度按 `expected_min_messages` 门槛判定）|
| 教师仪表盘 | P2 | 可视化学生数据，支持评分复核；阶段完成率与门槛同步 |
| 定时数据同步 | P2 | 从 Dify API 拉取日志，弥补非代理对话的数据缺失 |
| 平台配置运行时可改 | P2 | admin 在前端改 Dify API 地址 / Key 即时生效 |
| 批量录入学生 | P2 | CSV 严格事务（学号/姓名必填，密码默认 = 学号） |

## 2. 技术可行性分析

### 2.1 Dify API 能力评估

| 能力 | Dify 支持 | 我方使用方式 | 可行 |
|------|----------|-------------|------|
| 用户标识 | `user` 参数（任意字符串） | 生成 `dify_edu_{uuid}` 作为唯一标识 | ✅ |
| 流式对话 | SSE (Server-Sent Events) | Django `StreamingHttpResponse` 代理 | ✅ |
| 会话管理 | `/v1/conversations` 列表接口 | 同步到本地 `ConversationSession` 表 | ✅ |
| 消息历史 | `/v1/messages` 按会话查询 | 同步到本地 `ChatLog` 表 | ✅ |
| 会话变量 | Conversation Variables（持久化） | 在 Dify Workflow 中配置阶段变量 | ✅ |
| 文件上传 | `/v1/files/upload` | 代理上传接口 | ✅ |
| Token 统计 | `metadata.usage`（SSE `message_end` 事件） | 提取并存储 | ✅ |

**关键发现**：Dify 公开 API **完全覆盖** 本项目所有需求，无需修改 Dify 平台代码，零侵入对接。

### 2.2 阶段追踪可行性

**方案：Dify 会话变量 + SSE 事件解析（双重保障）**

| 检测方式 | 原理 | 可靠性 | 前提条件 |
|----------|------|--------|----------|
| **会话变量**（主要） | 教师在 Workflow 中用变量赋值器设置 `current_stage`，后端从 `node_finished`/`workflow_finished` 事件的 `outputs` 提取 | 高 | 教师需在 Dify 中正确配置 |
| **SSE 事件**（辅助） | 解析 `node_started`/`node_finished` 事件，根据节点标题匹配阶段 | 中 | 节点命名需规范 |

**结论**：技术上完全可行。会话变量机制是最可靠的方式，已在生产环境验证。即使教师暂未配置会话变量，系统仍可通过参与度维度评分。

### 2.3 Agent 类型的特殊性

本智能体是 **Agent** 类型（非 Chatflow），SSE 事件差异如下：

| 特性 | Chatflow 类型 | Agent 类型（本项目） |
|------|--------------|---------------------|
| 文本事件 | `text_chunk` | `agent_message` |
| 思考过程 | 无 | `agent_thought` |
| 消息完成 | `message` 事件 | `message_end` 事件 |
| Token 位置 | `message.usage` | `message_end.metadata.usage` |

**已适配**：代码同时兼容两种类型的事件格式。

### 2.4 流式模式限制

```
响应模式: 仅 streaming（不支持 blocking）
```

**影响与应对**：

| 影响 | 应对措施 |
|------|----------|
| Django WSGI 不支持流式 | 开发环境可用 `runserver`（异步线程），生产环境使用 Daphne/Uvicorn |
| 前端需消费 SSE 流 | 使用 `@microsoft/fetch-event-source` 替代原生 `EventSource`（支持 POST） |
| 网络中断数据丢失 | 后端在流中断时仍保存已接收的部分数据 |
| httpx 流式 API 差异 | 必须使用 `client.stream()` 上下文管理器 |

### 2.5 SQLite 适用性

| 场景 | 适用 | 说明 |
|------|------|------|
| 开发环境 | ✅ | 单用户，无并发问题 |
| 小规模教学（< 50 学生） | ✅ | 并发写入可控 |
| 大规模部署（> 200 学生） | ❌ | 建议切换 PostgreSQL |

### 2.6 三级用户层级实现可行性（v2.0 新增）

| 子需求 | 实现方式 | 可行 |
|--------|---------|------|
| 角色枚举 | `User.role ∈ {student, teacher, admin}` | ✅ |
| 教师独占其班级 | `SchoolClass.teacher` ForeignKey + view 层过滤 `teacher=request.user` | ✅ |
| 班级唯一性 | DB UNIQUE `(major_id, name)` | ✅ |
| 学生必属班级 | `User.school_class` FK PROTECT | ✅ |
| 关闭公开注册 | 删除 `/api/auth/register/` 路由与 Register.vue | ✅ |
| 软删除 | `is_active=False` + 接口默认过滤 | ✅ |
| 首次改密 | `must_change_password=True` + 前端登录后引导 | ✅ |
| Dify 配置运行时可改 | `DifyConfig` 单例 + admin 专属接口 | ✅ |
| 批量导入 | CSV 严格事务 + 表头/编码兼容 | ✅ |

**结论**：所有层级、权限、流程的关键约束都能用 Django ORM + DRF 干净表达，无需引入额外权限框架。

## 4. 数据库设计合理性

### 4.1 核心实体关系

```
Major ──1:N── SchoolClass
SchoolClass.teacher ──→ User[role=teacher]      (一个班级唯一教师)
SchoolClass ──1:N── User[role=student]          (学生必属班级)
User[role=student] ──1:1── DifyUserMapping      (仅学生绑定 Dify)
User ──1:N── ConversationSession ── 学习会话
ConversationSession ──1:N── ChatLog ── 逐条对话日志（核心分析数据源）
LearningWorkflow ──1:N── WorkflowStage ── 教学流程定义（含 expected_min_messages）
ConversationSession ──1:1── StudentScore ── 评分结果
StudentScore ──1:N── ScoreDetail ── 各阶段评分明细

DifyConfig (单例)     ── admin 在 /admin/dify-config 实时编辑
ScoringConfig (单例)  ── teacher 在 /teacher/settings 编辑
```

### 4.2 关键设计优势

- **Dify 与本地 ID 分离**: `dify_conversation_id` 和 `dify_message_id` 与自增 ID 分离，避免 ID 冲突
- **冗余字段**: `total_messages` / `total_tokens` 定期同步更新，减少查询开销
- **JSON 字段**: `completed_stages` / `inputs_data` 使用 JSON 文本存储，灵活扩展
- **三级 PROTECT 链**: SchoolClass→teacher、User→school_class、Major→SchoolClass 均为 PROTECT，避免误删传染
- **单例配置表**: `DifyConfig` / `ScoringConfig` 用 pk=1 强制单行，运行时可改、无需重启

## 5. 评分引擎可行性

### 5.1 四维度评分模型

| 维度 | 权重 | 数据来源 | 计算方式（v2.0 复核）|
|------|------|----------|----------|
| 阶段完成度 | 40% | `ChatLog.stage_key` 聚合计数 | 完成阶段数 / 总阶段数；阶段"完成" = 该阶段消息数 ≥ `WorkflowStage.expected_min_messages` |
| 流程遵循度 | 25% | `ChatLog.stage_key` 顺序 | 与预期顺序的匹配度 |
| 时间投入 | 15% | `Session.started_at → ended_at` | 实际耗时 / 预期耗时；低于 `min_session_minutes` 判 0 |
| 参与度 | 20% | `ChatLog` 消息计数 | 实际消息数 / 预期消息数 |

> v2.0 起，阶段"完成"判定从 v1.0 的"至少 1 条用户消息"统一改为按 `expected_min_messages` 门槛；同样的门槛也用于教师仪表盘的"各阶段完成率"（采用平均进度口径）。教师在 `/teacher/settings` 调整门槛后，可点"重算全部评分"批量刷新历史评分。

### 5.2 仪表盘阶段完成率口径（v2.0 新增）

仪表盘上的"各阶段完成率"采用 **平均进度（per-session cap）** 口径，**与"评分引擎的阶段完成度"不同但同源**：

```
rate = Σ_session min(该阶段消息数, expected_min_messages) / (expected_min_messages × 总会话数)
```

- 评分引擎是按"达标 / 不达标"的二值聚合，用于打分
- 仪表盘是按"消息数 / 门槛"的连续比例，用于宏观可视化
- 两者都依赖同一份 `expected_min_messages`，门槛改后两者同步联动

### 5.3 评分结果示例（已实测）

```
场景：学生完成1条对话（无阶段标记）

阶段完成度: 0.00  (无阶段标记)
流程遵循度: 0.00  (无阶段标记)
时间投入:   0.00  (小于5分钟)
参与度:    14.29  (1条消息 / 预期7条 × 100 × 0.20)
自动总分:   2.86
等级:      需改进
```

### 5.4 评分优化方向

1. **配置阶段标记后**：Dify 会话变量配置好后，阶段完成度维度可正常评分
2. **教师权重调整**：通过 `ScoringConfig` 单例 + `/teacher/settings` 实时调整
3. **门槛与重算**：教师改门槛后点"重算全部评分"批量刷新历史评分（`POST /api/teacher/scoring/recalculate-all/` 或 CLI `python manage.py recalculate_all_scores`）
4. **自然语言评估**：未来可引入 LLM 对回答质量进行语义评估

## 9. 最终结论与建议

### 9.1 总结论

**本项目在技术上完全可行。** Dify 平台提供的公开 API 足以支撑所有业务需求；v2.0 在 v1.0 的基础上落地了完整的"管理员 → 教师 → 班级 → 学生"组织结构，并把平台配置/评分参数都做成运行时可改的单例，覆盖了真实教学场景的日常运维需求。

### 9.2 关键技术决策回顾

| 决策 | 选择 | 理由 |
|------|------|------|
| 聊天UI | **自定义前端**（非 iframe） | 数据流可控，日志实时完整 |
| 阶段追踪 | **Dify 会话变量** | 最可靠，Dify 原生支持 |
| 流式处理 | **httpx `client.stream()`** | httpx 官方推荐的 SSE 消费方式 |
| 数据库 | **SQLite（开发）/ PostgreSQL（生产）** | SQLite 零配置适合开发，生产需 PostgreSQL |
| 前端 SSE | **`@microsoft/fetch-event-source`** | 支持 POST 请求和自定义 header |
| UI 库（v2.0） | **Element Plus + 自定义 token** | 业务页面快速搭建，深色侧边栏配商务风 |
| 平台配置 | **DB 单例 + admin 独占接口** | 运行时可改、无需重启、权限收窄 |
| 账号创建 | **自顶向下、关闭公开注册** | admin → teacher → 学生，外部无法横向注入 |
| 阶段完成判定 | **按 `expected_min_messages` 门槛** | 评分与仪表盘统一口径，教师可调 |
| 删除策略 | **软删 `is_active=False`** | 保留聊天/评分历史 |
